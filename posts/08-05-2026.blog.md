# Data Planning with DDD and ERD
After the wireframing stage, where I synthesised the core functions of the website, I used DDD and ERD to help me understand the needed data in each page, as well as the relationships this data has, how it benefits from each other and triggers the functions of the web app. While the DDD identified the necessary data needed throughout all the pages of the web app, the ERD showed how these tables affected each other.

## DDD: View-to-Data
 |View | Data Needed | User Action|    
 |-----|-------------|------------|
 |Home |current context, latest pain, saved items, recommended logs/threads | review recovery state|
 |Context |body areas, stages, goals, current limitation, preferences | update recovery context|
 |Recovery |stage guidance, timeline, related logs, exercise ideas | understand current recovery phase|
 |Log Form |current context, tags, pain before/after, symptoms | create recovery log| 
 |Explore |filters, similar users, logs, discussions | discover relevant community content|
 |Post Details |selected log/thread, replies, similar content | reply, save, report|
 |Account |context, latest pain, saved items | review personal summary|

## DDD Takeaways
I used the table above to help me understand possible data needs across our prototype in the making of the DDD plan. Through this table, I was able to understand the key requirements of each page's interaction. An example is the Homepage would need the 'recovery context', 'latest log updates', and 'recommended discussions'. The data becomes layered as the 'recovery context' needed in the Homepage would also need separate data inputs of 'body area', 'stages', etc.

This indicates that stages in the user flow like 'setting recovery context' and 'submitting recovery logs' are the most important areas of data within the web app that update and change an array of data across pages Home, Explore, Recovery and Exercise Guide.

## Key Decisions
With this in mind, in the DDD, I made a key decision to separate the 'recovery context' from the 'recovery logs'. The context will show the user's starting injury profile, while the recovery logs show the reflection of the user's progress. This means pain should not be stored statically within context. Instead, the latest pain update should be linked with the recovery log data, since it changes through recovery stages.

## ERD Planning
Following the DDD, I used the ERD below to plan out how tables would interact across the prototype. This ERD included main entities Users, Recovery contexts, Recovery logs, Discussions, Replies, Saved items, and Reports. I employed lookup tables for Body areas, Recovery stages, and Activity goals as they could range in inputs, so Users, Logs and Discussions can be matched consistently.

## Key Points of the ERD
A notable relationship I got from the ERD is that a user can have multiple recovery logs while only having one recovery context, supporting a stable profile while allowing updates to shift data across the site over time. Logs and discussions are both classified by body area, recovery stage, and activity goal, which validates the Explore page's discovery algorithm. Saved items and reports were also included as a way to strengthen the community function of the site by making the Explore page more interactive.

## Reflection
These diagrams made a significant impact in refining the project scope, breaking down the unique interaction between pages and data to lay out the foundation for an adequate execution of technical development. Narrowing down the core functions beyond the conceptual phase of the wireframes kept the prototype within scope and prioritised recovery systems and community over a broad medical advice system.
