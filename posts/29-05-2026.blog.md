## About our Web App
Through the project we developed a recovery community hub prototype that helps users reflect on recovery while facilitating social engagement. Users get to set recovery contexts to track their progress, reflect on them with structured logs, and find relevant community experiences while managing privacy preferences. They are also able to interact with community posts by archiving or reporting.

## Performance and Technical Behaviour
To test the quality of our code implementation, we ran and passed manual server checks that consisted of walkthroughs of sample user flows, as well as terminal-based checks that included:
- npm run build
- npm run lint
- npm test

`npm run build` verified that the web app did not have any TypeScript or module errors. Furthermore, `npm run lint` helped identify syntax, formatting and convention issues like 'camelCase' naming conventions for functions. Last but not least, `npm test` tested model behaviour, such as the latest pain on the Homepage being derived from recovery logs, Explore page returning related logs and discussions, and save, report and privacy actions writing to or blocking the correct data.

Additionally, the model tests ensured Explore page returned related results in logs, discussions and user profiles based on recovery contexts and search filters. This gave evidence that the discovery features within our web app was supported by database logic instead of being interface placeholders. They also verified that invalid user inputs would be rejected and be provided with comprehensive feedback messages to make the web app more reliable and responsible for its interactions.

The automated testing, along with manual web app walkthroughs, showed that the site was responsive at local testing scale with a smaller SQLite database. However, this does not prove performance under larger public use, and future testing would require larger datasets, and multi-user scenario testing, where our community integration will be challenged in terms of its processing efficiency with filtering and recommendations.

## UX and Accessibility 

### Navigation and Interaction Flow
A major change in the site's UI during development was the move away from a grid-like presentation. The final layout focused more on the core functions of each page, with side rails for supporting information and navigation. Account features were separated from main recovery flow for more streamlined interactions. Moreover, Explore pages were enhanced with working filters, and save/report actions. The Recovery page became less cluttered with information, and focused towards user progress and inputs. Finally, feedback messages and actionable empty states refined the web app by providing clear system responses.

### Accessibility Checks
To evaluate the overall UI, each page is put through WAVE evaluation, manually cross-matched with AA WCAG standards, as well as keyboard navigation and screen-reader testing. WAVE and WCAG were able to reaffirm visual contrasts and accessibility for interface navigation making sure the web app presented a cohesive visual flow. While keyboard navigation, screen reader along with remaining requirements under WCAG AA guidelines sharpened the accessibility of our web app functions, like being able to navigate the site through keyboard navigation and screen readers, and pausing moving components like the social feed through focused scrolling.

## Functional Requirements

Our starting concept adopted a user flow that would take user recovery context and provide advice based on secondary research to the user. Users would take this advice and document their experience through community posts online. However the scope of the concept was quite far fetched, as it needed proper trust systems and professional backing to provide valid medical advice. Furthermore, the web app also would need a complicated recommendation algorithm to provide appropriate references from just their short recovery context. 

This led to our shift towards a more community-centred and personal design. The final web app instead acts as an online journal and tracker, as well as a social platform for recovery related topics. Specifically, users can receive suggested posts based on their recovery context through matching tags, and submit logs to their progress after trying advice stated in community discussions. To strengthen the readiness of the prototype, we also included privacy features and settings, that lets users ensure a secure experience, especially when the app stores personal medical data.

This way the web app was made to be more feasible in terms of execution by taking away the research based feedback that would include complex algorithmic code for personalised recommendations, and it became more feasible logistically when applied to real-world scope, as we don't have a clear way to solidify the website as a trusted source of medical advice.

 ## Highlights in Planning and Limitations
Though we went through a lot of reflection while making our web app, it still leaves room for improvement, and could definitely be refined further in a product-ready iteration. The Explore matching system is simple and uses preset tags to return similar posts. In a more developed version, our search engine should use weighted matching on keywords and tags for a more nuanced discovery experience. Report systems would also need to be actionable by moderators through designated dashboards, rather than just stored as records. Furthermore, privacy guidelines within the prototype show the limitations of the current iteration, which lacks corporate-backed authentication and data handling. Most importantly, an addition of multi-user testing processes across both desktop and mobile platforms would significantly improve its readiness. These limitations highlight a clearer path towards improvement in future development.

## Conclusion
Our web app development informed me about the many layers and considerations needed in web design. I learned that route architecture, database interactions and visual styling need to complement the same design goals for a reliable and user-centred product. Examples like error states or discussion creation helped solidify the interactions between user and system. I also learned that reframing can be necessary, a different purpose isn't a worse purpose in most cases, and our choice to make our final app a community-backed reflective journal made it more feasible for us, and safer for the users.
