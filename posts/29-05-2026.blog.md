## About our Web App
Through the project we were able to develop a web app prototype of a recovery community hub that helps users reflect on recovery while facilitating social engagement. Its core purpose allows users to set recovery contexts to track their progress, reflect on them with structured log submissions, and find relevant community experiences while adhering to the users' privacy preferences. Private or public users are able to interact with community posts by archiving or reporting.

## Performance and Technical Behaviour
To test the quality of our code implementation, we ran and passed manual server checks that consisted of thorough walkthrough of sample user-flows, as well as terminal-based checks that included:
- npm run build
- npm run lint
- npm test

'npm run build' verified that our web app didn't have any type errors within data fields, as well as any module errors that could have been created during refactoring. Furthermore, 'npm run lint' helped identify syntax, formatting and convention issues like 'camelCase' naming conventions for functions. Last but not least, 'npm test' tested the model behaviour like the latest pain from homepage was derived from recovery logs, explore page returning related logs and discussions, as well as save, report and privacy actions writing to or blocking the correct data.
Additionally, the model test ensured explore page returned related results in logs, discussions and user profiles based on recovery contexts and search filters. This gave evidence that the discovery features within our web app was supported by database logic instead of being interface placeholders. And finally, verifying that invalid user inputs would be rejected and be provided with comprehensive feedback messages to make the web app more reliable and responsible for its interactions.

The automated testing along with manual web app walkthroughs showed that the site was smooth and responsive under local testing scales, with a smaller SQLite database. However, this does not prove performance under larger public use, and future testing would require larget datasets, and multi-user scenario testing, where our community integration will be challenged in terms of its processing efficiency with filter searching and reccomendations.

## UX and Accessibility 

### Navigation and Interaction Flow
A major change in the site's UI during development was the move away from a grid-like presentation. The final layout focused more on the core functions of each page and with side rails for supporting information and navigation.  Specific changes across each page for example are Account features being separated from main recovery flow for more streamlined interactions. Moreover, explore pages were enhanced with added working filters, and post interactions like save, and reports that made the experience more socially engaging. The recovery page became less cluttered with information, and focused towards user progress and inputs. Other familiar ideas that needed some time to develop are the feedback messages for both negatives and positives outcomes, and actionable empty states that refined the web app by facilitating helpful interactions between the system and user.

### Accessibility Checks
To evaluate the overall UI, each page is put through WAVE evaluation, manually cross-matched with AA WCAG standards, as well as keyboard navigation and screen-reader testing. WAVE and WCAG were able to reaffirm visual contrasts and accessibility for interface navigation making sure the web app presented a cohesive visual flow. While keyboard navigation, screen reader along with remaining requirements under WCAG AA guidelines sharpened the accessibility of our web app functions, like be able to navigate the site that included semantic labels without mouse clicks, and pause the moving social feed in the homepage of the web app through focus states.

## Functional Requirements

Our starting concept adopted a user flow that would take user recovery context and provide advice based on secondary research to the user. Users would then take on this advice and document their experience through community posts online. However the scope of the concept was quite far fetched, as it needed proper trust systems and professional backing to provide valid medical advice. Furthermore, the web app also have needed a deep and complicated recommendation algorithm to provide appropriate references from just their short recovery context. 

This led to our shifts towards a more community centred and personal design. The final web app instead acts as an online journal and tracker, as well as a social platform for recovery related topics. Specifically, users can receive suggested posts based on their recovery context through matching tags, and submit logs to their progress after trying advice stated in community discussions. To strengthen the readiness of the prototype, we also included privacy features and settings, that lets users ensure a secure experience, especially when the app stores personal medical data.

This way the web app was made to be more feasible in terms of execution by taking away the research based feedback that would include complex algorithmic code for personalised recommendations, and it became more feasible logistically when applied to real-world scope, as we don't have a clear way to solidify the website as a trusted source of medical advice.

 ## Highlights in Planning and Limitations
Though we were able to go through a lot of reflective process during the making of our web app, it still leaves room for improvement, and could definitely be refined further in a product-ready iteration. Our matching system present in the explore page is simple and uses things set tags to return similar posts, in a more developed version, our search engine should have a weighted matching system on keywords and not tags alone, that would provide a more nuanced  discovery experience of the web app. Also, staying on the explore page, in a professional web app, report systems would need to be actionable by moderators through designated dashboards, and isn't just stored as memory for a specific profile. Furthermore, privacy guidelines within the prototype acts as a sample for the limitations it provides, missing real corporate backed authentication and data handling. Most importantly, since the web app was mostly tested manually within the group, an addition of multi-user testing processes across both desktop and mobile platforms would significantly improve its readiness. These limitations are all valuable feedback that we can gather information from that highlights a clearer path towards improvement in future development.

## Conclusion
Our web app development informed me about the many layers and considerations needed in web design. Having a strong architecture of backend to frontend components, from routes, and database interactions to visual styling is crucial for a design to mesh in terms of functionality and concept, translating consistent output from backend logic to frontend outcome makes the web app more reliable and user-centred. Examples like error states or discussion creation plays a huge part solidifying the interactions between user and system. I also learned that reframing isn't as tragic as it seems, a different purpose isn't a worse purpose in most cases, and our choice to surround our final web app around a community backed reflective purpose made the app more feasible for us, and safer for the users.
