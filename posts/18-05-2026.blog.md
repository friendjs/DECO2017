# Goal
This blog details my evaluation plans, methods, and goals for my design. Our aim right now is to create a web app that can run reliably, support the main user flow, with a variety of functions working together that handle the database cleanly.

# Evaluation Plan
## User-flow Testing
The main user flow that I will be testing throughout my development is:
- Login
- Users set recovery context
- Read details about their recovery progress
- Explore similar experiences from others
- Submit a recovery log that updates the user's progress

As the core of the web app, this line of interaction connects a recovery tracking journal to a community hub that invites social engagement and discovery of similar experiences. 

Within this ideal user flow, there are specific areas across the pages of the web app that I am looking to test the effectiveness of. Would users be able to understand the recovery snapshot in the Homepage? Does recovery detail state body area, stage, and goal clearly? How well does the Explore page return relevant discussion related to the user's context? And finally, does the log form feel useful as a journal for progress? These can be tested through user testing, while I also plan to test it myself by working through the user flow.

## Technical Testing
Aside from the user flow testing that can be vague or subjective depending on the testing participant, I will also look to trial other areas of the code technically.

During development and coding, test commands like:
- npm run lint
- npm run test
- npm run build

will help check the validity of our code, if the model logic behaves correctly, as well as the build of our TypeScript.

## Accessibility 
As an essential for all web apps, the final prototype will need to pass accessibility requirements like adequate contrasts, clean layout, visible labels, responsiveness, etc. These various aspects can be tested through things like the WCAG principles, platforms like WAVE or Lighthouse, or manual processes like tab navigation, or screen reader testing.

These tests culminate to refine and perfect our prototype to a professional standard that signifies its accessibility to more users.

## Risks 
As we look to progress further towards development of our specific design, we need to factor in the risks and limitations associated inherently within our design.

This could be for example the validity of medical advice. Due to our lack of qualification, we need to make sure users understand our recommendations come from past user experiences and not professional advice. Further risks could be privacy risks associated with health data, or also authentication risks when users are logging in. Along with this, some limitations of the current concept to consider are a basic search engine and basic report options.

## Future Improvements
Reflecting on the design's risks and limitations, some improvements for the future could be clear privacy controls, a detailed report page with types of offences, authenticated login, and declaration of qualification that lets users know that the web app is a community hub, not a professional advisor. I also acknowledge that as the prototype develops, obstacles are certain to appear, in which improvements to things like modularity will need revising for a more comprehensive structure even if users don't see it.
